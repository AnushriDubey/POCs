    @Bean
    @Order(2)

        // Standard security for the login form itself

            .authorizeHttpRequests(authorize -> authorize
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults());

Private Key: Stays on your server. It is used to "sign" the JWT (like a wax seal on an envelope).

Public Key: Shared with the world via the /.well-known/jwks.json endpoint. Anyone can use this to verify the seal wasn't tampered with, but they can't create their own seal.

2. Why you got the code_challenge Error

The Engineering Flow of PKCE:

Client creates a secret string (Verifier) and hashes it (Challenge).

Client sends the Challenge to the Server.

Server gives a temporary code.

Client sends the original secret Verifier + the code back to the Server.

Server hashes the Verifier; if it matches the original Challenge, it issues the Token.

How does the Resource Server know the Public Key has updated?
In a professional engineering setup, the Resource Server doesn't just download the key once and forget it forever. It uses two main strategies:

1. The "Cache & TTL" Strategy (Standard)
Most Spring Boot Resource Servers use a library (like nimbus-jose-jwt) that caches the keys it gets from the jwks_uri.

TTL (Time To Live): The library is usually configured with a refresh interval (e.g., every 5 to 60 minutes).

The Logic: Even if it has a key, it will periodically re-fetch the jwks_uri to see if a new key has been added to the set.

2. The "On-Demand Refresh" Strategy (Reactive)
This is the most robust method. JWTs often include a field in their header called kid (Key ID).

The Scenario: Your Auth Server rotates keys. A user presents a new JWT signed with Key B.

The Resource Server's Logic:

It looks at its cache and sees it only has Key A.

Instead of rejecting the token immediately, it says: "Wait, I don't recognize Key B. Let me check the jwks_uri one more time right now."

It fetches the updated JWK Set, finds Key B, updates its cache, and then validates the token.

Why the JWK Set is a "Set" (The keys: [] array)
If you look at http://localhost:8080/oauth2/jwks, you will see it returns a list of keys, not just one. This allows for Graceful Key Rotation:

Phase 1: Auth Server generates a new key (Key B) but keeps the old one (Key A) in the list.

Phase 2: For a period of time, the server publishes both keys. Resource servers download the set and can now validate tokens signed by either.

Phase 3: Once all old tokens (signed by Key A) have expired, the Auth Server removes Key A from the list.

A. The ID Token (The Badge)
In a full OIDC flow, the /oauth2/token endpoint returns two tokens instead of one:

Access Token: Used to call the API (Resource Server).

ID Token: A specialized JWT that contains the user's profile (name, email, etc.). The 3rd-party app (Postman/Frontend) reads this to say "Welcome, Admin!" on the screen.

B. The UserInfo Endpoint
A full-fledged OIDC server provides a standard /userinfo endpoint.

The Flow: If an app has a valid Access Token but needs more details (like the user's profile picture or phone number), it calls this endpoint. The Auth Server looks at the token and returns the user's private profile data.

C. Standardized Scopes
OIDC introduces "Reserved" scopes that every server must understand:

openid: (Mandatory) Tells the server to treat this as an OIDC request.

profile: Requests access to name, nickname, and birthdate.

email: Requests access to email and email_verified status.

3. The Engineering "Handshake" with OIDC
When you use "Full-Fledge" OIDC, the Discovery step we discussed earlier becomes the foundation:

Discovery: The Client (Postman) hits /.well-known/openid-configuration.

Auth Request: The Client sends the user to /authorize with scope=openid profile.

Token Exchange: The Client receives an Access Token and an ID Token.

Validation: The Client (or Resource Server) fetches the Public Keys from /oauth2/jwks to verify that the tokens haven't been tampered with.

4. Why is this better for your Resource Server?
If you weren't using OIDC/JWT:

Your Resource Server would receive a random string like XY123.

It would have to call the Auth Server: "Hey, what does XY123 mean? Who is this?" (This is called Introspection).

This makes your API slow because every request requires a second network call.

With OIDC/JWT:

Your Resource Server receives a signed JSON object.

It sees {"sub": "admin", "roles": ["USER"]}.

It checks the digital signature locally.

Result: It trusts the data immediately without asking the Auth Server for help.

Goal	OAuth2 Only	OIDC (Full-Fledge)
Main Output	Access Token (Opaque string)	Access Token + ID Token (JWT)
User Identity	Not included	Included in id_token and /userinfo
Configuration	Manual / Hard-coded	Automatic via .well-known
Resource Server	Must call Auth Server to verify	Verifies locally using Public Keys

A 403 Forbidden error on a Resource Server is actually a "good" sign—it means your Authentication worked (the server knows who you are), but your Authorization failed (you don't have the right "Permission" or "Scope").

Here is the engineering checklist to find the disconnect between your Auth Server and Resource Server.

1. Check the "Scope" in the Token
The most common cause is that the token you generated doesn't actually contain the read or write scope required by your SecurityFilterChain.

How to Debug:

Copy the JWT you are using in Postman.

Go to jwt.io.

Paste the token and look at the Payload.

Look for the scope field. It should look like this:

JSON
{
  "sub": "admin",
  "scope": "openid profile read",
  "iss": "http://localhost:8080",
  ...
}
If the scope is empty or missing read: Your Postman request to the Auth Server didn't ask for the scope, or your RegisteredClient in the Auth Server doesn't have .scope("read") configured.

2. The "SCOPE_" Prefix Match
In your Resource Server code, Spring Security automatically takes the scope claim from the JWT and adds the prefix SCOPE_.

The Logic:

If JWT has "scope": "read", Spring sees the authority SCOPE_read.

If your code has .hasAuthority("read"), it will fail with a 403.

Fix: Ensure your config uses the prefix:

Java
.requestMatchers("/api/user/**").hasAuthority("SCOPE_read")

The Resource Server is very strict about the iss claim. It must be a 100% character-for-character match with the issuer-uri in your application.yml.

Auth Server says: http://localhost:8080

Resource Server config says: http://127.0.0.1:8080

Result: 403 Forbidden (or 401). Even though the IP is the same, the strings must match exactly.
